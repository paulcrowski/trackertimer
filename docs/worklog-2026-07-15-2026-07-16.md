# Worktimer worklog — July 15–16, 2026

This is a plain-language handoff of what we worked through in ChatGPT over the
last two days. It records the problem, the decision, what is already deployed,
and the one practical step still needed on a user's computer.

## English

### What started the work

Convex showed about **1.16 GB of Database I/O**, while the actual database
storage was only about **1.76 MB**. Those are different measurements: the first
is traffic caused by reads and writes, not the amount of data sitting in the
database.

The main suspect was the desktop helper. It was checking activity too often
and could continue making requests even when the timer was not meaningfully
running. That is a bad fit for a product that should eventually support many
people: a tiny request repeated all day becomes a large bill at scale.

### The decision we made

The timer remains manual and is still the source of truth. The helper is an
optional assistant, not a silent monitoring system.

The new flow is:

1. The helper samples the foreground app and window context locally every five
   seconds.
2. It keeps only compact, consecutive context blocks in a small JSON buffer on
   the computer. Users do not install SQLite or configure a local database.
3. The browser page and helper talk through a localhost bridge. The page keeps
   a short lease alive only while an active timer session is running.
4. While the timer is stopped, paused, or the page disappears, the helper does
   not upload activity data.
5. Private domains and sensitive titles are masked before a summary leaves the
   computer.
6. The helper sends a compact checkpoint roughly every 15 minutes and one
   final summary at STOP. It does not send every five-second sample to Convex.
7. Convex stores summaries idempotently by session and revision, so retries do
   not create duplicate rows.

We chose 15-minute checkpoints instead of hourly-only uploads so a crashed
computer does not lose an entire workday. The payload remains small, and the
final STOP summary closes the session cleanly.

### What was implemented

- A localhost helper bridge on port `32145`.
- Local sampling, buffering, privacy masking, lease heartbeats, and explicit
  start/stop capture commands.
- A new `desktopSessionSummaries` Convex table and `/api/desktop/summary` route.
- Server-side privacy protection and idempotent summary ingestion.
- Production configuration moved to the new Convex deployment
  `combative-bullfrog-707`; the old project's historical 1 GB was not copied.
- English and Polish README documentation explaining the local-first model and
  the fact that the normal timer works without the helper.

### Verification

The local CI gate passed: typecheck, lint, formatting, tests, build, and local
smoke. The public frontend returned HTTP 200. The deployed summary endpoint
returned HTTP 401 without a helper key, which confirms that the route exists
and is protected rather than missing.

### What to do on a real computer

A fresh helper starter must be downloaded from the production app and started
again. An older helper process does not automatically gain the new localhost
bridge. After restarting it, check that the production page says the helper is
connected, run one short timer session, then press STOP and confirm that one
compact summary arrives.

No old 1 GB needs to be migrated for this demo. The safer cost decision is to
leave the old project alone and test the new compact flow with fresh data.

## Polski

### Od czego zaczęliśmy

Convex pokazał około **1,16 GB Database I/O**, ale sama baza miała tylko około
**1,76 MB**. To dwie różne rzeczy: pierwsza liczba mówi o ruchu odczytów i
zapisów, a nie o tym, ile danych leży w bazie.

Najbardziej podejrzany był desktopowy helper. Sprawdzał aktywność zbyt często i
mógł nadal wykonywać zapytania, kiedy timer nie był naprawdę uruchomiony. Przy
jednej osobie wygląda to niewinnie, ale przy 100 osobach takie małe zapytania
powtarzane przez cały dzień robią się kosztowne.

### Jak to rozwiązaliśmy

Timer nadal uruchamia i zatrzymuje użytkownik. To on jest źródłem prawdy.
Helper jest opcjonalną pomocą, a nie cichym systemem monitoringu.

Nowy przepływ wygląda tak:

1. Helper lokalnie sprawdza aktywną aplikację i tytuł okna co 5 sekund.
2. Trzyma krótkie, kolejne bloki kontekstu w małym buforze JSON na komputerze.
   Użytkownik nie musi instalować SQLite ani żadnej lokalnej bazy.
3. Strona rozmawia z helperem przez localhost. Krótki lease jest odnawiany
   tylko podczas aktywnej sesji timera.
4. Gdy timer jest zatrzymany, spauzowany albo strona znika, helper nie wysyła
   danych aktywności do Convex.
5. Prywatne domeny i wrażliwe tytuły są maskowane jeszcze przed wysłaniem
   podsumowania.
6. Do Convex trafia mały checkpoint mniej więcej co 15 minut i jedno końcowe
   podsumowanie po STOP. Nie wysyłamy każdej próbki co 5 sekund.
7. Podsumowania są zapisywane idempotentnie według sesji i rewizji, więc retry
   nie tworzy duplikatów.

Wybraliśmy checkpoint co 15 minut zamiast wysyłki tylko raz na godzinę, żeby
awaria komputera nie kasowała całego dnia pracy. Dane nadal są małe, a STOP
zamyka sesję jednym końcowym podsumowaniem.

### Co zostało wdrożone

- Lokalny most helpera na porcie `32145`.
- Lokalne próbki, buforowanie, maskowanie prywatności, heartbeat lease oraz
  jawne komendy start/stop zbierania.
- Nowa tabela Convex `desktopSessionSummaries` i endpoint
  `/api/desktop/summary`.
- Dodatkowa ochrona prywatności po stronie serwera oraz bezpieczne retry bez
  duplikowania podsumowań.
- Produkcja została przełączona na nowy deployment Convex
  `combative-bullfrog-707`; starego 1 GB nie przenosiliśmy.
- README po angielsku i po polsku opisują teraz model local-first oraz to, że
  zwykły timer działa bez helpera.

### Sprawdzenie

Lokalny `npm run ci` przeszedł: typy, lint, formatowanie, testy, build i lokalny
smoke test. Publiczna strona zwróciła HTTP 200. Nowy endpoint podsumowań bez
klucza zwrócił HTTP 401, czyli istnieje i jest chroniony — nie jest brakującą
trasą 404.

### Co zostało praktycznie do zrobienia

Trzeba pobrać świeży starter helpera z produkcyjnej strony i uruchomić go
ponownie. Stary proces nie dostaje automatycznie nowego mostu localhost. Po
restarcie strona powinna pokazać połączenie z helperem. Następnie warto zrobić
jedną krótką sesję, nacisnąć STOP i sprawdzić, czy wpada jedno małe
podsumowanie.

Starego 1 GB nie ma sensu migrować w tym demie. Najbezpieczniej kosztowo jest
zostawić stary projekt bez zmian i testować nowy, oszczędny przepływ na świeżych
danych.
